这是一个 Flutter 跨平台 App（hachimi-app）的重大版本更新，设计文档位于: docs/all-new-hachimi-dnd/

请你先完整阅读该目录下的所有文档，建立全局理解后再开始审计。

## 任务：对设计方案进行全面、严苛的上线前审计

请以"即将开始编码实现"为前提，用最挑剔的技术审查视角，完成以下五轮独立审查。每轮审查请单独输出，不要混在一起。

### 第一轮：顶层设计一致性校验

回溯文档中的顶层设计原则和架构目标（如果有明确声明的话），逐一检查：
- 各子模块的设计是否真正服务于顶层目标，还是出现了偏移？
- 各模块之间的职责边界是否清晰，有无职责重叠或真空地带？
- 有没有哪个设计决策与顶层原则相矛盾？

### 第二轮：逐章细节成熟度评估

对每一份文档、每一个章节，评估其"可编码程度"：
- 是否已经精确到可以直接作为开发依据？还是仍停留在"方向性想法"层面？
- 数据结构、状态流转、API 契约、事件模型等是否有明确定义？
- 边界情况、异常处理、降级策略是否有考虑？
- 请用一个三级标度标注每个章节：✅ 可编码 / ⚠️ 需补充 / ❌ 仅为方向

### 第三轮：胶水编程 vs 造轮子 识别

Flutter 生态和 Dart 社区有大量成熟方案。请逐一检查设计中：
- 哪些能力已有成熟的 Flutter/Dart package 可以直接采用或封装？（列出推荐的 package）
- 哪些地方看起来像在"从零实现"已有的成熟方案？
- 哪些地方确实需要自研（无现成方案或现有方案不满足需求），请说明理由

### 第四轮：开发卡点预判

假设明天就开始编码，预判可能遇到的卡点：
- 框架层面的限制或已知坑
- 状态管理复杂度是否会失控？
- 性能热点在哪里？
- 测试策略是否可行？哪些部分难以测试？
- 各模块之间的集成顺序和依赖关系是否清晰？

### 第五轮：缺失清单与优先级建议

综合以上四轮，输出一份结构化的"开工前必须补齐的清单"：
- 按 P0（阻塞开发）/ P1（影响质量）/ P2（可后补）分级
- 每一项给出具体的补充建议，而不是笼统地说"需要细化"

## 输出格式要求

- 引用具体文档名和章节位置
- 批评要具体、可操作，避免泛泛而谈
- 如果某个部分设计得好，也请明确肯定，说明好在哪里
