## 1. 产品重新定位：从工具到觉知

### 1.1 旧定位（v2.x）

> Hachimi 是一款把「养猫」和「专注计时」深度融合的习惯养成 App。

**问题诊断**（用梁宁的话说）：这是一个「匪兵甲」级别的功能价值产品——习惯打卡 + 番茄钟 + 虚拟宠物。功能可替代，市面上 Finch、Habitica、Forest 都能做到。猫咪虽然有情绪价值的潜力，但目前只停留在「装饰」层面，没有和用户的真实情感需求产生深度连接。

### 1.2 新定位（v3.0）

> **Hachimi 是一个帮你在每天 5 分钟里「觉知自己」的数字伴侣。**
>
> 猫咪不再只是习惯追踪的奖励机制，而是你内心宇宙里陪你一起成长的生命。  
> 你照顾猫咪，猫咪照顾你。  
> 每天记录一点光，日积月累，点亮内心宇宙。

### 1.3 定位转变的本质

用梁宁的第一性原理来思考：这个品类（自我管理/成长类 App）的**第一性**是什么？

不是「打卡」，不是「追踪」。**第一性是「觉知」——帮人看见自己，看见自己的模式、情绪和变化。**

播客中白丽丽的体验完美印证了这一点：她用了两个月韧性手册，最大的收获不是「养成了几个习惯」，而是**「知道了什么真正让我开心」**——她发现「临水吹风」是自己的充电方式，发现开心事可以分成三类。这种觉知，才是让产品从「工具」变成「不可替代的伙伴」的关键。

梁宁说：**「功能价值是工具，情绪价值是玩具。工具用完即走，玩具让人想留。」** Hachimi 要从工具变成玩具，从玩具变成生命档案。

## 2. 方法论框架：梁宁《真需求》的应用

### 2.1 产品价值公式

梁宁的核心公式：

```
产品价值 = 功能价值 + 情绪价值 + 资产价值
```

应用到 Hachimi v3.0：

| 价值维度 | v2.x 现状 | v3.0 目标 | 对应的用户需求 |
|---------|----------|----------|-------------|
| **功能价值** | 习惯打卡 + 专注计时 | 习惯设计（四法则引导）+ 烦恼管理 + 周/月计划 + 每日时间轴 | 效率需求：帮我管好时间和习惯 |
| **情绪价值 — 保障感** | 几乎没有 | 烦恼处理器（写下烦恼→想解法→追踪状态）、大脑减负 | 对抗焦虑：把烦恼从脑袋里卸载出来 |
| **情绪价值 — 愉悦感** | 猫咪陪伴（浅层） | 每日一点光、三个幸福时刻、猫咪情感反应、模式发现 | 对抗枯燥：让每天都有值得记录的瞬间 |
| **情绪价值 — 彰显性** | 无 | 成长历程可视化、年度回顾报告 | 内在自我确认：我在变好 |
| **资产价值** | 无 | 个人成长档案（不可替代的生命记录） | 投资需求：时间越长越有价值 |

### 2.2 关系的四个次第

梁宁在第十一章讲到用户关系的四个层次：**需要 → 喜欢 → 认同 → 归属**。

Hachimi 的关系递进设计：

| 次第 | 用户状态 | 产品策略 | 对应功能 |
|------|---------|---------|---------|
| **需要** | 我需要管理习惯和情绪 | 提供确定性的功能价值 | 习惯打卡、专注计时、周计划 |
| **喜欢** | 我喜欢这只猫、喜欢每天的仪式感 | 创造情绪价值的愉悦感 | 猫咪互动、每日一点光、完成动画 |
| **认同** | 我认同「觉知自己、善待自己」的价值观 | 让用户通过使用发现自我模式 | 三个幸福时刻、模式分析、周回顾 |
| **归属** | 这里记录了我的生命，不可替代 | 积累资产价值，提高迁移成本 | 成长档案、年度回顾、猫咪成长历程 |

### 2.3 KANO 模型：特性与属性的映射

梁宁在第六章强调：**特性是自身视角（我做了什么），属性是用户视角（用户感受到什么）**。

我们必须用 KANO 模型来分类每个功能的属性，确定 MVP 的优先级（详见第 5 节）。

### 2.4 四法则：习惯设计方法论

播客中提到的「四法则」（看得见、想去做、易上手、有奖励），本质上是 James Clear《原子习惯》四法则的中国化表达。Hachimi 的习惯创建流程必须内置这个方法论：

| 法则 | 中文 | 英文 | 在 App 中的实现 |
|------|------|------|---------------|
| 第一法则 | 看得见 | Make it obvious | 主屏 Widget、智能通知、猫咪提醒 |
| 第二法则 | 想去做 | Make it attractive | 猫咪期待动画、习惯串联提示 |
| 第三法则 | 易上手 | Make it easy | 每次交互 ≤ 2 分钟、一键打卡 |
| 第四法则 | 有奖励 | Make it satisfying | 猫咪庆祝、彩纸动画、打卡勾选的满足感 |

### 2.5 场景匹配：品类的第一性

梁宁在第八章反复强调：**没有强场景的好产品，就是一直在地上爬的虫子。**

播客中韧性手册的成功，核心在于它精准嵌入了三个具体场景：月底、周末、睡前。每次只要 5 分钟。门槛低到「不需要动用『坚持』这个词」。

Hachimi v3.0 的场景设计完全对标这三个时间节点（详见第 4 节）。

### 2.6 避免「貂丁」

梁宁在第二章警告：很多产品「自己说得牛气冲天，看起来市场空白，做得很费资源，结果用户却很尴尬」。

Hachimi v3.0 的反貂丁原则：

1. **每天只要 5 分钟**——这是产品的生命线。一旦变重，用户就会放弃
2. **猫咪不能喧宾夺主**——用户来是为了觉知自己，不是为了养猫。猫咪是载体，不是目标
3. **AI 要克制**——梁宁说情绪价值的第一性是「关怀」而不是「操纵」。AI 不催促，而是在用户回顾时安静地呈现发现
4. **不堆砌功能**——MVP 只做三个场景的核心体验，做透做深

## 3. 用户人设

梁宁在第七章说：**用户画像 ≠ 用户人设。画像是统计学的，人设是心理学的。一个人的长项就是他的需求。**

### 3.1 核心人设：「努力但容易自我怀疑的成年人」

| 维度 | 描述 |
|------|------|
| **年龄** | 25-45 岁 |
| **自我期待** | 想变得更好：更自律、更健康、更了解自己 |
| **核心痛点** | 买过很多手账和日记本，但大多写了几页就放一边了。坚持不下去不是因为不想，而是因为门槛太高、反馈太慢 |
| **隐性需求** | 不知道「什么真正让自己开心」，忙了一天觉得没什么特别开心的事。烦恼压在心里，半夜醒来翻来覆去 |
| **行为特征** | 手机重度用户，睡前刷手机，容易被短视频吞噬时间 |
| **情感特征** | 喜欢可爱的东西（猫、卡通），但不想被当小孩子看待 |
| **消费心理** | 愿意为「让自己变好」的东西付费，但需要看到明确的效果。心理账户中有「自我投资」这一项 |

### 3.2 人设公式（梁宁第七章）

```
用户人设 = 自我期待（想成为的人）+ 外部压力（现实约束）+ 内在恐惧（害怕的事）
```

- **自我期待**：成为一个有觉知、自律、内心丰盈的人
- **外部压力**：工作忙、家庭琐事多、时间碎片化
- **内在恐惧**：害怕自己一直这样浑浑噩噩下去，害怕不够好

**产品切入点**：在「自我期待」和「外部压力」之间找到最小可行的支点——每天 5 分钟，足够简单到不会被外部压力挤掉，又足够有效到能触及自我期待。

### 3.3 非目标用户

- 需要严格时间管理系统（GTD、OKR）的项目经理型用户
- 需要社交打卡、互相监督的外向型用户
- 只想玩养成游戏、不关心自我成长的用户

## 4. 场景设计：三幕结构

完全对标播客中韧性手册的三个场景，用数字化能力做增强。

### 4.1 第一幕：月初仪式（每月 1 次，约 5 分钟）

**触发时机**：每月 1 号，或用户主动进入「月度规划」页面

**场景还原**（来自播客）：
> 月底，我会花 5 分钟时间，列出下个月想做的事情、想养成的习惯。

**功能设计**：

| 模块 | 说明 | 交互方式 |
|------|------|---------|
| 本月目标 | 写下本月想做的 3-5 件事 | 文本输入，每条一行 |
| 小赢挑战 | 选择本月想养成的一个习惯 | 从已有习惯中选择或新建 |
| 四法则设计 | 引导用户用四法则设计这个习惯 | 四个勾选项 + 可选文本说明 |
| 30 天打卡格 | 每天完成小赢挑战后打勾 | 日历格子，点击打勾 |
| 月度奖励 | 完成 30 天打卡后的自我奖励 | 文本输入（「如果我完成 30 天，我就奖励自己 ___」） |

**猫咪互动**：月初时猫咪会换上一个小装饰（比如一朵花、一条围巾），代表新月份的开始。

### 4.2 第二幕：周日复盘（每周 1 次，约 5 分钟）

**触发时机**：每周日晚上，推送提醒；或用户主动进入「周度回顾」页面

**场景还原**（来自播客）：
> 周日，花 5 分钟，写下下周必须干的事、重要但不紧急的事、自己很想干的事。  
> 回顾这周过得怎么样，写下「本周的三个幸福时刻」「这周我想感恩谁」「这一周我学到了什么」。

**功能设计**：

**A. 周计划（Plan 侧）**

| 模块 | 说明 | 交互方式 |
|------|------|---------|
| 给本周的自己一句话 | 自我激励 | 单行文本输入 |
| 四象限安排 | 紧急重要 / 重要不紧急 / 紧急不重要 / 想做的事 | 四个区域，每区域可添加条目 + 预估时长 |
| 烦恼处理器 | 写下烦恼 + 想到的解法 | 列表形式，每条包含：烦恼描述、解法、日期 |

**B. 周回顾（Reflect 侧）**

| 模块 | 说明 | 交互方式 |
|------|------|---------|
| 三个幸福时刻 | 本周最幸福的三个瞬间 | 三个卡片，可附带 emoji 或标签 |
| 感恩 | 这周我想感恩谁 | 单行文本 |
| 学到了什么 | 本周的收获 | 单行文本 |
| 烦恼状态更新 | 对之前写下的烦恼做分类 | 三种状态按钮：自然消失 / 自己搞定 / 还在 |

**猫咪互动**：周回顾完成后，猫咪会用它独特的方式「总结」这一周（基于 AI，用猫咪视角写 1-2 句）。

### 4.3 第三幕：睡前一点光（每天 1 次，约 2 分钟）

**触发时机**：用户设定的睡前时间推送提醒；或用户主动打开

**场景还原**（来自播客）：
> 睡前，花 5 分钟，在时间轴上写下今天主要干的三四件事情，然后写「每日一句」，回顾今天有什么开心的事情。

**功能设计**：

| 模块 | 说明 | 交互方式 |
|------|------|---------|
| 今日时间轴 | 回顾今天的主要事项 | 简化时间轴（6:00-24:00），可快速标记 3-4 件事 |
| 心情选择 | 今天整体心情 | 5 个 emoji（很开心 / 开心 / 平静 / 低落 / 很低落） |
| 今日一点光 | 今天有什么让你开心的事？ | 单行文本输入（支持一句话，也支持多写） |
| 习惯打卡 | 今天的小赢挑战完成了吗？ | 一键打勾 |

**猫咪互动**：
- 打开时猫咪在打瞌睡，看到你来了会醒过来蹭蹭你
- 写完「一点光」后，猫咪给出一个小小的反应（头蹭、发出呼噜声），不是 AI 废话，只是一个温暖的动作
- 全部完成后，猫咪蜷成一团睡着了，配上一句话：「晚安，明天见」

**核心设计原则**：这是全 App 最轻、最核心的交互。**最少只需要一个操作**——选一个心情 emoji，写一句话。30 秒就能完成。

## 5. KANO 属性分析与 MVP 边界

用梁宁第六章的 KANO 模型，将所有特性按用户属性分类：

### 5.1 必备属性（没有就不满意，MVP 必须做）

| 特性 | 用户预期 | MVP 实现标准 |
|------|---------|------------|
| 离线可用 | App 在无网络环境下 100% 可用 | 沿用现有 SQLite SSOT 架构 |
| 习惯打卡 | 能够创建习惯、每日打卡、看到连续天数 | 沿用现有功能 |
| 专注计时 | 能够进行番茄钟/正计时 | 沿用现有功能 |
| 猫咪基础互动 | 猫咪有名字、有性格、会陪伴 | 沿用现有功能 |
| 数据安全 | 不会丢失我的记录 | 沿用现有备份机制 |

### 5.2 期望属性（有就满意，没有就不满意，MVP 核心交付）

| 特性 | 用户预期 | MVP 实现标准 |
|------|---------|------------|
| 睡前「一点光」 | 每天能快速记录今天的开心事 | 心情 emoji + 一句话文本框 + 习惯打卡勾选 |
| 周回顾三个幸福时刻 | 每周回顾幸福瞬间 | 三个文本卡片 + 感恩 + 学习 |
| 烦恼处理器 | 把烦恼写下来，想解法，追踪状态 | 列表 CRUD + 三种状态标记 |
| 猫咪睡前互动 | 猫咪在我写完后有温暖的反应 | 3-5 个预设动画（蹭蹭、呼噜、打瞌睡） |
| 今日时间轴 | 快速回顾今天做了什么 | 简化时间轴 + 快速标记 |

### 5.3 魅力属性（有就惊喜，没有也不会不满意，MVP 后期或 v3.1 做）

| 特性 | 说明 | 优先级 |
|------|------|--------|
| AI 模式发现 | 连续记录 4 周后，AI 帮你发现幸福时刻的模式 | v3.1 |
| 月计划 + 四法则引导 | 完整的月度计划设计功能 | v3.1 |
| 周计划四象限 | 紧急/重要矩阵式周计划 | v3.1 |
| 年度回顾报告 | 自动生成的年度成长报告 | v3.2 |
| 猫咪冒险 | 猫咪根据你的记录去冒险，带回和你相关的小礼物 | v3.2 |
| 主屏 Widget | 在手机主屏显示猫咪和今日进度 | v3.1 |

### 5.4 反向属性（做了反而减分，绝对不做）

| 特性 | 为什么不做 |
|------|----------|
| 社交分享/排行榜 | 觉知是私密的，社交压力会破坏安全感 |
| 催促式通知（「你今天还没打卡！」） | 梁宁说情绪价值的第一性是「关怀」不是「操纵」 |
| 断连惩罚（中断连续天数的负面后果） | 播客体验者说「有半个月不熬夜也是进步」——产品要包容不完美 |
| 强制每日打开 | 门槛越低越好，偶尔忘记不应该有惩罚 |

### 5.5 MVP 边界总结

**MVP 做什么**（期望属性 + 必备属性）：

1. ✅ 保留现有全部功能（习惯打卡、专注计时、猫咪养成、AI 日记/聊天、配饰商城、成就系统）
2. ✅ 新增：睡前「一点光」功能（心情 + 一句话 + 打卡）
3. ✅ 新增：周回顾功能（三个幸福时刻 + 感恩 + 学到了什么）
4. ✅ 新增：烦恼处理器（写烦恼 + 想解法 + 状态追踪）
5. ✅ 新增：今日时间轴（简化版，快速标记 3-4 件事）
6. ✅ 新增：猫咪睡前互动动画（3-5 个预设动画）
7. ✅ 新增：底部导航增加「觉知」Tab

**MVP 不做什么**（魅力属性，留给后续版本）：

- ❌ AI 模式发现
- ❌ 月度计划完整版
- ❌ 周计划四象限
- ❌ 年度回顾报告
- ❌ 猫咪冒险系统
- ❌ 主屏 Widget

## 6. 用户体验地图（完整版）

### 6.1 全局导航结构（v3.0）

底部 Tab 从 4 个扩展为 5 个：

| Tab | 图标 | 页面 | 说明 |
|-----|------|------|------|

## 8. 数据模型（新增表）

### 8.1 SQLite 新增表

在现有 `sqflite` 数据库中新增以下表：

```sql
-- 每日一点光记录
CREATE TABLE daily_lights (
    id TEXT PRIMARY KEY,               -- UUID
    date TEXT NOT NULL UNIQUE,          -- 'YYYY-MM-DD' 格式，每天一条
    mood INTEGER NOT NULL,              -- 0=veryHappy, 1=happy, 2=calm, 3=down, 4=veryDown
    lightText TEXT,                     -- 一点光文本内容（可为空）
    tags TEXT,                          -- JSON 数组，如 '["工作成就","学习"]'
    timelineEvents TEXT,               -- JSON 数组，如 '[{"time":"09:00","text":"开会","emoji":"💼"}]'
    habitCompletions TEXT,             -- JSON 对象，如 '{"habitId1":true,"habitId2":false}'
    createdAt TEXT NOT NULL,           -- ISO 8601 时间戳
    updatedAt TEXT NOT NULL            -- ISO 8601 时间戳
);

-- 周回顾记录
CREATE TABLE weekly_reviews (
    id TEXT PRIMARY KEY,               -- UUID
    weekId TEXT NOT NULL UNIQUE,        -- 'YYYY-WXX' 格式，如 '2026-W12'
    weekStartDate TEXT NOT NULL,        -- 本周一的日期 'YYYY-MM-DD'
    weekEndDate TEXT NOT NULL,          -- 本周日的日期 'YYYY-MM-DD'
    happyMoment1 TEXT,                 -- 幸福时刻 1
    happyMoment1Tags TEXT,             -- JSON 数组
    happyMoment2 TEXT,                 -- 幸福时刻 2
    happyMoment2Tags TEXT,             -- JSON 数组
    happyMoment3 TEXT,                 -- 幸福时刻 3
    happyMoment3Tags TEXT,             -- JSON 数组
    gratitude TEXT,                     -- 感恩
    learning TEXT,                      -- 学到了什么
    catWeeklySummary TEXT,             -- 猫咪 AI 周总结
    createdAt TEXT NOT NULL,
    updatedAt TEXT NOT NULL
);

-- 烦恼处理器
CREATE TABLE worries (
    id TEXT PRIMARY KEY,               -- UUID
    description TEXT NOT NULL,          -- 烦恼描述
    solution TEXT,                      -- 解法（可为空）
    status TEXT NOT NULL DEFAULT 'ongoing',  -- 'ongoing' | 'resolved' | 'disappeared'
    resolvedAt TEXT,                    -- 解决时间
    createdAt TEXT NOT NULL,
    updatedAt TEXT NOT NULL
);

-- 觉知相关成就追踪
CREATE TABLE awareness_streaks (
    id TEXT PRIMARY KEY DEFAULT 'singleton', -- 只有一行
    currentLightStreak INTEGER NOT NULL DEFAULT 0,  -- 当前一点光连续天数
    longestLightStreak INTEGER NOT NULL DEFAULT 0,  -- 最长一点光连续天数
    totalLightDays INTEGER NOT NULL DEFAULT 0,      -- 总记录天数
    totalWeeklyReviews INTEGER NOT NULL DEFAULT 0,  -- 总周回顾次数
    totalWorriesResolved INTEGER NOT NULL DEFAULT 0, -- 总解决烦恼数
    lastLightDate TEXT,                              -- 上次记录日期
    updatedAt TEXT NOT NULL
);
```

## 11. PDCA 迭代记录

### 第一轮 PDCA

**Plan（计划）**：
基于梁宁《真需求》的价值公式和播客中韧性手册的三个场景，将 Hachimi 从纯功能价值的习惯追踪工具，升级为覆盖功能价值 + 情绪价值 + 资产价值的「觉知伴侣」。MVP 聚焦三个场景：月初仪式、周日复盘、睡前一点光。

**Do（执行）**：
完成了完整的 PRD 编写，包含：方法论框架、用户人设、场景设计、KANO 属性分析、用户体验地图（5 条完整旅程）、MVP 功能规格（含 Widget 级别的代码规格）、数据模型、技术实现指引、分阶段计划。

**Check（检查）**：

发现的问题：

1. **月初仪式在 MVP 中的定位模糊**：虽然在场景设计中描述了月初仪式，但在 MVP 边界中将其归为「魅力属性」放到了 v3.1。这和「三幕结构」的完整性矛盾。需要决定：是在 MVP 中做一个极简版月计划，还是完全推迟？
2. **通知策略缺失**：整个 PRD 没有详细说明推送通知的策略。播客中白丽丽的使用体验严重依赖「场景嵌入」，而数字产品的场景嵌入很大程度依赖通知。
3. **标签体系的价值未充分说明**：标签看起来像一个小功能，但实际上它是未来 AI 模式发现的数据基础。需要明确标签的设计意图。
4. **觉知统计页面未展开**：「统计」Tab 提到新增「觉知统计」，但没有详细说明展示什么内容。

**Act（改进）**：

**改进 1：月初仪式降级为 MVP 极简版**

在 MVP 中增加月计划的最小可行版本：只保留「本月小赢挑战」和「30 天打卡格」两个模块，去掉「本月目标」和「四法则引导」。这样三幕结构完整，但实现成本很低。

具体来说，在「觉知」Tab 的「今日」子页面顶部，当检测到是新月份且还没有设置月度挑战时，显示一个提示卡片：「设置本月的小赢挑战？」→ 点击后弹出 BottomSheet → 选择一个习惯 → 自动生成 30 天打卡格。

**改进 2：补充通知策略**

新增通知设计规范：

| 通知类型 | 触发条件 | 时间 | 文案示例 | 频率 |
|---------|---------|------|---------|------|
| 睡前一点光 | 用户设定的睡前时间 | 每晚一次 | 「猫咪在等你说晚安 🐱」 | 每日 |
| 周日复盘 | 周日 20:00（可调） | 每周一次 | 「这周辛苦啦，来回顾一下吧 ✨」 | 每周 |
| 月初仪式 | 每月 1 日 10:00 | 每月一次 | 「新的一月，设个小目标？🌱」 | 每月 |
| 温柔提醒 | 连续 3 天未记录 | 第 3 天 20:00 | 「猫咪有点想你了…… 🥺」 | 最多每周 1 次 |

绝对不做的通知：
- ❌ 「你今天还没打卡！」（催促式）
- ❌ 「你的连续天数即将中断！」（恐吓式）
- ❌ 每天多于 1 条通知（打扰式）

**改进 3：明确标签的战略意图**

标签不是随便加的装饰。它是「AI 模式发现」（v3.1 魅力属性）的数据基础。当用户记录了 30+ 天的「一点光」并打了标签后，AI 可以告诉用户：

> 「你的幸福时刻中，65% 和 #户外 有关，30% 和 #家人 有关。看起来出门走走和家人在一起是你最好的充电方式。」

这就是白丽丽发现「临水吹风是我的充电方式」的数字化版本。

因此在 MVP 中，标签的选择是必须做的（虽然标为「可选」让用户不觉得负担），但后端必须存好这个数据。

**改进 4：补充觉知统计页面**

在「统计」Tab 中新增「觉知」子 Tab（与现有「习惯」「成就」并列），显示：

- 一点光连续天数（当前 / 最长）
- 本月心情日历（emoji 热力图）
- 心情分布饼图（过去 30 天的心情比例）
- 烦恼处理统计（已解决 / 进行中）
- 幸福时刻标签云（出现频率最高的标签放大显示）

### 第二轮 PDCA

**Plan（计划）**：
基于第一轮的四项改进，细化 MVP 的完整规格，确保 Claude Code 可以直接执行。

**Do（执行）**：
将四项改进整合到 PRD 主体中（已在上文对应位置更新），并补充以下遗漏项。

**Check（检查）**：

再次审视整个 PRD，发现以下问题：

1. **空状态设计缺失**：首次使用觉知功能时，历史页面、统计页面都是空的。空状态的设计直接影响用户第一印象。
2. **数据导出/备份未提及**：梁宁说的「资产价值」依赖于用户对数据所有权的信心。如果用户担心数据丢失，资产价值就不成立。
3. **「不完美也没关系」的设计理念需要更明确地贯穿产品**：播客中白丽丽说「打了一半的卡，转念一想，有半个月不熬夜也是进步」。这个理念必须在 UI 中体现。
4. **猫咪互动的情感颗粒度不够**：目前猫咪只在「一点光完成后」有睡前动画。当用户心情是「低落」或「很低落」时，猫咪的反应应该不同。

**Act（改进）**：

**改进 5：空状态设计**

| 页面 | 空状态文案 | 空状态插图 |
|------|----------|----------|
| 今日一点光（未记录） | 「今天还没有记录光哦，什么时候都可以来 ✨」 | 猫咪趴在窗台看月亮 |
| 周回顾（未回顾） | 「慢慢来，等你准备好了再回顾 🐱」 | 猫咪翻开一本小本子 |
| 觉知历史（无记录） | 「每一天的光都值得被记住」 | 猫咪仰望星空 |
| 烦恼处理器（无烦恼） | 「没有烦恼？太好了！🎉」 | 猫咪开心打滚 |
| 觉知统计（数据不足） | 「再记录几天，就能看到有趣的发现了 📊」 | 猫咪研究一张图表 |

空状态插图使用现有 PixelCatSprite 渲染，不需要额外资源。

**改进 6：数据导出功能（MVP 极简版）**

在「我的」Tab → 设置 → 数据管理中，新增「导出觉知数据」选项：

```dart
/// 导出为 JSON 文件，包含：
/// - 所有 daily_lights 记录
/// - 所有 weekly_reviews 记录
/// - 所有 worries 记录
/// - awareness_streaks 统计
///
/// 文件名：hachimi_awareness_{YYYY-MM-DD}.json
/// 通过 Share Sheet 分享
```

**改进 7：「不完美也没关系」设计原则**

具体的 UI 表达：

- **连续天数中断时**：不显示红色警告，而是显示「中断了也没关系，今天又是新的开始 🌱」
- **周回顾不完整时**：三个幸福时刻允许只填 1-2 个，保存按钮不因未填满而 disable
- **月度打卡未满时**：月底显示「这个月打了 X/30 天，每一天都算数 ✨」而不是「未完成」
- **烦恼未解决时**：「还在」状态不是负面的，显示为 🟡 而非 ❌，文案为「还在的小烦恼——慢慢来」

**改进 8：心情自适应猫咪互动**

根据用户选择的心情，猫咪的反应不同：

| 用户心情 | 猫咪反应 | 文案 |
|---------|---------|------|
| 😄 很开心 | 猫咪跳起来蹦蹦跳跳 | 「铲屎官今天好开心，本猫也开心！🎉」 |
| 🙂 开心 | 猫咪蹭蹭你 | 「记录了今天的第 N 束光 ✨」 |
| 😌 平静 | 猫咪安静地趴在旁边 | 「平静的一天也很好呢 🍃」 |
| 😔 低落 | 猫咪轻轻靠过来 | 「不开心的日子，本猫陪着你 💛」 |
| 😢 很低落 | 猫咪蜷在你身边不走 | 「没事的，本猫哪都不去 🫂」 |

这种心情自适应的猫咪互动，是梁宁所说「关怀」而非「操纵」的具体体现。猫咪不会说「明天会更好」这种空洞的正能量，而是用行动表达「我在这里，陪着你」。

## 附录 B：关键设计原则速查表

| 原则 | 来源 | 在产品中的体现 |
|------|------|-------------|
| 每天只要 5 分钟 | 播客 | 核心交互 ≤ 60 秒可完成 |
| 门槛低到不需要「坚持」 | 播客 | 最少操作 = 1 次心情选择 + 1 句话 |
| 功能价值是工具，情绪价值是玩具 | 梁宁·第二章 | 猫咪从装饰升级为情感连接载体 |
| 关怀而非操纵 | 梁宁·第十六章 | AI 不催促，猫咪不惩罚 |
| 特性是自己的，属性是用户的 | 梁宁·第六章 | 用 KANO 模型验证每个功能的用户属性 |
| 没有强场景就是在地上爬的虫子 | 梁宁·第八章 | 三幕结构嵌入月末/周末/睡前 |
| 不做貂丁 | 梁宁·第二章 | 不堆砌功能，不追求全面，做透核心场景 |
| 不完美也没关系 | 播客 | 中断不惩罚，少打卡也认可进步 |
| 需要→喜欢→认同→归属 | 梁宁·第十一章 | 从功能吸引→情感连接→价值观认同→资产沉淀 |
| 用一个字概括好产品：美 | 梁宁·第十六章 | 每个界面、每个动画都要自然、温暖、不多不少 |
