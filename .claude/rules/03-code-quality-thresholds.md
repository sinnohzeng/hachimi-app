# 开发质量红线（硬性阈值）

> 所有文件必须严格遵守以下阈值，无例外。

## 强制阈值

| 指标 | 红线 | 说明 |
|------|------|------|
| 单文件行数 | ≤ 800 行 | 超过即拆分为多个文件 |
| 单函数行数 | ≤ 30 行 | 超过即提取子函数 |
| 嵌套层数 | ≤ 3 层 | 超过即使用 early return 或提取方法 |
| 分支数量 | ≤ 5 个 / 函数 | 见下方分级标准 |

### 分支数量分级标准

游戏逻辑（骰子检定、属性计算、场景分支等）天然包含较多条件路径。原阈值 ≤ 3 过于严格，参考 McCabe 圈复杂度（Cyclomatic Complexity）行业标准（严格项目 5-7，通用标准 10）调整为分级制：

| 等级 | 分支数 | 含义 | 动作 |
|------|--------|------|------|
| 🟢 安全 | ≤ 3 | 理想状态 | 无需额外关注 |
| 🟡 审查触发 | 4-5 | 需要 Review 确认合理性 | Code Review 时重点关注，确认无法通过查表/策略模式简化 |
| 🔴 硬性上限 | 6-10 | 必须重构或提供书面理由 | 提交前必须尝试重构；若确属不可简化的游戏逻辑，在代码注释中说明理由 |
| ⛔ 绝对天花板 | > 10 | 禁止合并 | 无例外，必须拆分 |

> **注**：此调整源于 DnD 游戏逻辑需求——骰子检定结果分支、职业特性判定、场景事件路由等场景中，3 分支上限会导致过度碎片化的函数拆分，反而降低可读性。非游戏逻辑代码仍建议保持 ≤ 3 的理想状态。

## 执行规则

1. **新建文件**：必须在编写时即满足全部阈值
2. **修改文件**：触及的函数/类必须在修改后满足阈值
3. **Code Review**：任何超阈值的代码不允许合并
4. **分支审查**：出现 4 个及以上分支时，必须在 Review 中说明无法进一步简化的理由（参见 01-persona-coding-principles.md 第四章）

## 常见修复手段

- **文件过长**：按职责拆分（一个类一个文件、widget 提取到 components/）
- **函数过长**：提取 private helper、使用 builder 模式
- **嵌套过深**：guard clause（early return）、提取方法、flatMap
- **分支过多**：Map 查表、策略模式、枚举 + switch expression
